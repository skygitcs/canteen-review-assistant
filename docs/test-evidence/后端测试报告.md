## 4. 测试过程摘要

### 4.1 数据库初始化

通过 MySQL 命令行执行：

```sql
source D:/26spring/Android/APP/canteen-review-assistant/backend/src/main/resources/db/schema.sql;
use thu_canteen;
source D:/26spring/Android/APP/canteen-review-assistant/backend/src/main/resources/db/seed.sql;
```

初始化后数据库包含 12 张表：

```text
announcements
canteen_windows
canteens
consumption_records
crowd_reports
```

示例数据中包含桃李园、紫荆园、听涛园等食堂，以及麻婆豆腐、牛肉拉面、宫保鸡丁等菜品。

### 4.2 服务启动与接口文档

在 `backend/` 目录下执行：

```powershell
mvn spring-boot:run
```

日志显示 Tomcat 已启动在 `8080` 端口，后端服务启动成功。随后访问 Swagger UI，可以看到接口分组和接口列表，说明 OpenAPI 文档生成正常。

### 4.3 基础冒烟测试

说明基础接口、JWT 登录态、拥挤度上报和收藏接口均可用。

### 4.4 管理员审核链路

管理员审核链路按以下流程完成：

```text
普通用户注册登录
-> 普通用户提交新菜品
-> 菜品进入 PENDING 待审核状态
-> 注册管理员候选账号
-> MySQL 将该用户 role 设置为 ADMIN
-> 管理员重新登录获取 ADMIN Token
-> 管理员查看待审核列表
-> 管理员审核通过
-> 系统生成正式菜品
-> 普通菜品搜索接口可以查询到该菜品
```

关键接口结果如下：

| 步骤 | 接口或操作 | 关键结果 |
|---|---|---|
| 普通用户登录 | `POST /api/auth/login` | 返回 `role = USER` 和 JWT Token |
| 提交菜品 | `POST /api/dishes/submissions` | 返回 `auditStatus = PENDING` |
| 设置管理员 | `UPDATE users SET role='ADMIN' ...` | 查询结果显示 `role = ADMIN` |
| 管理员登录 | `POST /api/auth/login` | 返回 `role = ADMIN` 和 JWT Token |
| 查看待审核 | `GET /api/admin/submissions` | 返回刚提交的待审核菜品 |
| 审核通过 | `POST /api/admin/submissions/{id}/approve` | 返回正式菜品，`status = APPROVED` |
| 查询菜品 | `GET /api/dishes?keyword=...` | 能查到审核后的正式菜品 |

### 4.6 负向测试

本轮负向测试用于验证后端在异常场景下能否正确拒绝非法请求，覆盖权限控制、审核驳回和参数校验。

#### 4.6.1 普通用户访问管理员接口

普通用户携带 `USER` Token 访问：

```text
GET /api/admin/submissions
```

后端返回：

```json
{
  "code": 403,
  "message": "permission denied",
  "data": null
}
```

说明普通用户无法访问管理员接口，RBAC 权限控制生效。

#### 4.6.2 管理员驳回菜品

普通用户提交用于驳回测试的新菜品后，管理员调用：

```text
POST /api/admin/submissions/{id}/reject
```

驳回接口返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": null
}
```

随后通过普通菜品搜索接口查询该菜品：

```text
GET /api/dishes?keyword=审核驳回测试菜品_161236
```

返回结果为：

```json
{
  "code": 0,
  "message": "ok",
  "data": []
}
```

说明被驳回的菜品没有进入正式菜品列表，`PENDING -> REJECTED` 审核分支生效。

#### 4.6.3 非法评分校验

普通用户提交非法评分：

```json
{
  "rating": 6,
  "content": "非法评分测试，rating 超过 5。"
}
```

后端返回：

```json
{
  "code": 400,
  "message": "rating: 最大不能超过5",
  "data": null
}
```

再次查询菜品详情后，评论数保持不变：

```text
beforeReviewCount = 1
afterReviewCount  = 1
```

说明非法评分没有写入数据库，也没有影响评分统计。

## 6. 发现的问题与说明

1. 原始测试日志中包含完整 JWT Token。正式提交或展示时建议使用截图，或对 Token 进行打码，避免把完整 Token 放入公开材料。
2. `application.yml` 已改为通过环境变量读取数据库连接信息，提交前仍需确认没有把个人本机密码写入配置文件或测试材料。
3. 部分新增对象接口刚返回时 `createdAt`、`updatedAt` 为 `null`，重新查询数据库后时间字段存在。原因是插入后返回的是内存对象，没有重新查询数据库自动填充后的记录，不影响本轮功能验证。
4. 当前后端采用统一响应格式，业务异常通常返回 HTTP 200，但响应体中通过 `code` 表示业务状态。例如 `code = 403` 表示无权限，`code = 400` 表示参数非法。测试时应同时检查响应体业务状态码。
