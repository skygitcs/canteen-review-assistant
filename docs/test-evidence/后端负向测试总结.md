# 后端负向测试总结

## 一、测试目的

本轮负向测试用于验证后端在异常场景下是否能够正确拒绝非法请求，重点检查权限控制、审核驳回逻辑和数据合法性校验是否生效。

前期正向测试已经验证了数据库初始化、后端启动、基础接口、JWT 鉴权、管理员审核通过、菜品正式展示和评论评分统计等主流程。本轮测试进一步补充异常场景，确保系统不仅能处理正常请求，也能对越权访问、审核驳回和非法评分等情况进行正确处理。

## 三、测试内容与结果

### 1. 普通用户访问管理员接口测试

#### 测试目的

验证普通用户是否会被禁止访问 `/api/admin/**` 管理员接口，从而确认后端权限控制逻辑是否生效。

#### 测试过程

首先注册并登录普通用户：

```powershell
$normalUsername = "negative_user_$(Get-Date -Format 'MMddHHmmss')"
```

登录后返回用户角色为：
```json
{
  "role": "USER"
}
```
随后使用该普通用户的 Token 访问管理员接口：
```text
GET /api/admin/submissions
```

#### 实际结果
接口返回：
```json
{
  "code": 403,
  "message": "permission denied",
  "data": null
}
```
测试脚本进一步解析统一响应体后输出：
```text
HTTP Status = 200
Business code = 403
Message = permission denied
[PASS] 普通用户访问管理员接口被拒绝，权限控制生效
```

#### 结论
该测试通过。普通用户无法访问管理员接口，后端权限控制生效。

需要注意的是，当前后端采用统一响应格式，业务错误仍然返回 HTTP 200，但响应体中的 `code = 403` 表示无权限访问。因此判断结果时应以业务状态码 `code` 为准，而不是只看 HTTP 状态码。

### 2. 管理员驳回菜品测试
#### 测试目的

验证管理员驳回用户提交菜品后，该菜品不会进入正式菜品列表，从而确认 `PENDING -> REJECTED` 审核分支是否正确。

#### 测试过程

首先注册管理员候选账号，并通过 MySQL 将其角色修改为 `ADMIN`：
```sql
UPDATE users SET role='ADMIN' WHERE username='$adminUsername';
SELECT id, username, role FROM users WHERE username='$adminUsername';
```
查询结果显示：
```text
role = ADMIN
```

管理员重新登录后返回：
```json
{
  "role": "ADMIN"
}
```
随后普通用户提交一个用于驳回测试的新菜品：
```json
{
  "name": "审核驳回测试菜品_161236",
  "auditStatus": "PENDING"
}
```
管理员通过待审核列表查询到该菜品，并调用驳回接口：
```text
POST /api/admin/submissions/{id}/reject
```

驳回请求体为：
```json
{
  "reason": "测试驳回：菜品信息不完整"
}
```
#### 实际结果
驳回接口返回：
```json
{
  "code": 0,
  "message": "ok",
  "data": null
}
```
随后通过普通菜品搜索接口查询该菜品：
```text
GET /api/dishes?keyword=审核驳回测试菜品_161236
```
返回结果为：
```json
{
  "code": 0,
  "message": "ok",
  "data": []
}
```

#### 结论

该测试通过。管理员驳回菜品后，该菜品没有进入正式菜品列表，普通用户无法通过菜品搜索接口查询到该菜品，说明审核驳回逻辑生效。

### 3. 非法评分测试
#### 测试目的

验证后端是否能够拒绝非法评分，防止评分范围错误的数据写入数据库并影响统计结果。

#### 测试过程

首先查询菜品 1 的评论数：
```powershell
$beforeDish = Invoke-RestMethod `
  -Method Get `
  -Uri "$BASE/api/dishes/1"

$beforeReviewCount = $beforeDish.data.base.reviewCount
```
测试前评论数为：
```text
beforeReviewCount = 1
```
随后提交非法评分：
```json
{
  "rating": 6,
  "content": "非法评分测试，rating 超过 5。",
  "imageUrl": "https://example.com/bad-review.jpg"
}
```
该请求调用接口：
```text
POST /api/dishes/1/reviews
```

#### 实际结果
后端返回：
```json
{
  "code": 400,
  "message": "rating: 最大不能超过5",
  "data": null
}
```
随后再次查询菜品 1 的评论数：
```text
beforeReviewCount = 1
afterReviewCount  = 1
```
#### 结论

该测试通过。非法评分 `rating = 6` 被后端拒绝，且评论数没有增加，说明非法数据没有写入数据库，也没有影响菜品评分统计。

## 五、测试结论

本轮负向测试通过。测试结果表明，后端不仅能够处理正常业务流程，也能够在异常场景下进行有效限制和校验。

具体来说：

1. 普通用户访问管理员接口时，后端返回业务状态码 `403`，说明管理员权限控制生效。

2. 管理员驳回用户提交的菜品后，该菜品不会进入正式菜品列表，普通菜品搜索接口返回空列表，说明审核驳回逻辑生效。

3. 用户提交非法评分时，后端返回业务状态码 `400`，且评论数未增加，说明评分数据校验和统计保护逻辑生效。

因此，后端在权限安全、内容审核安全和数据准确性方面均通过了本轮负向测试。结合前期的数据库初始化、服务启动、基础冒烟测试、管理员审核通过链路和评论统计测试，后端部分已经具备较完整的正向与负向测试证据。

---

## 六、注意事项

1. 当前后端采用统一响应格式，业务异常通常返回 HTTP 200，但响应体中会通过 `code` 表示具体业务状态。例如：
   - `code = 403` 表示无权限；
   - `code = 400` 表示参数非法。

2. 因此在测试时不能只根据 HTTP 状态码判断成功或失败，应同时检查响应体中的业务状态码。

3. 测试输出中包含 JWT Token。正式报告或截图中建议对 Token 长串进行打码，避免泄露认证信息。

4. 本轮负向测试已经覆盖权限控制、审核驳回和非法评分。后续如需进一步完善，可以继续补充重复收藏、重复评论、非法食堂/窗口关系、未登录访问用户接口等异常场景。

### 附录 C. 测试普通用户访问管理员接口

本测试用于验证普通用户是否会被禁止访问 /api/admin/** 管理员接口。
```powershell
$resp = Invoke-WebRequest `
  -Method Get `
  -Uri "$BASE/api/admin/submissions" `
  -Headers $normalHeader

$json = $resp.Content | ConvertFrom-Json

"HTTP Status = $($resp.StatusCode)"
"Business code = $($json.code)"
"Message = $($json.message)"

if ($json.code -eq 403) {
    Write-Host "[PASS] 普通用户访问管理员接口被拒绝，权限控制生效"
} else {
    Write-Host "[FAIL] 普通用户不应成功访问管理员接口"
    $json | ConvertTo-Json -Depth 10
}
```
预期结果：
```text
HTTP Status = 200
Business code = 403
Message = permission denied
[PASS] 普通用户访问管理员接口被拒绝，权限控制生效
```
说明：当前后端采用统一响应格式，业务异常通过响应体中的 `code` 表示。因此即使 HTTP 状态码为 200，只要业务状态码为 `403`，就表示权限拒绝成功。

### 附录 F. 测试管理员驳回菜品

本测试用于验证 `PENDING -> REJECTED` 审核分支是否正常，即管理员驳回后，菜品不应进入正式菜品列表。

#### 1. 普通用户提交用于驳回测试的菜品
```powershell
$rejectDishName = "审核驳回测试菜品_$(Get-Date -Format 'HHmmss')"

$rejectSubmissionBody = @{
    canteenId = 1
    windowId = 1
    name = $rejectDishName
    imageUrl = "https://example.com/images/reject-test.jpg"
    price = 20.00
    description = "用于测试管理员驳回流程。"
    spiceLevel = 1
    tags = @("测试", "驳回")
} | ConvertTo-Json

$rejectSubmission = Invoke-RestMethod `
  -Method Post `
  -Uri "$BASE/api/dishes/submissions" `
  -Headers $normalHeader `
  -ContentType "application/json" `
  -Body $rejectSubmissionBody

$rejectSubmission | ConvertTo-Json -Depth 10
```
预期返回中应包含：
```json
{
  "auditStatus": "PENDING"
}
```

#### 2. 管理员查看待审核列表
```powershell
$pending = Invoke-RestMethod `
  -Method Get `
  -Uri "$BASE/api/admin/submissions" `
  -Headers $adminHeader

$pending | ConvertTo-Json -Depth 10
```
获取刚提交的待审核菜品 ID：
```powershell
$rejectSubmissionId = ($pending.data | Where-Object { $_.name -eq $rejectDishName })[0].id
$rejectSubmissionId
```

#### 3. 管理员驳回该菜品
```powershell
$rejectBody = @{
    reason = "测试驳回：菜品信息不完整"
} | ConvertTo-Json

$rejectResult = Invoke-RestMethod `
  -Method Post `
  -Uri "$BASE/api/admin/submissions/$rejectSubmissionId/reject" `
  -Headers $adminHeader `
  -ContentType "application/json" `
  -Body $rejectBody

$rejectResult | ConvertTo-Json -Depth 10
```
预期返回：
```json
{
  "code": 0,
  "message": "ok",
  "data": null
}
```

#### 4. 查询正式菜品列表，确认驳回菜品不存在
```powershell
$encodedRejectDishName = [uri]::EscapeDataString($rejectDishName)

$rejectSearchResult = Invoke-RestMethod `
  -Method Get `
  -Uri "$BASE/api/dishes?keyword=$encodedRejectDishName"

$rejectSearchResult | ConvertTo-Json -Depth 10

$matched = $rejectSearchResult.data | Where-Object { $_.name -eq $rejectDishName }
$matched
```
预期结果：
```json
{
  "code": 0,
  "message": "ok",
  "data": []
}
```
且 `$matched` 为空，说明被驳回菜品没有进入正式菜品列表。

### 附录 G. 测试非法评分

本测试用于验证评分范围校验是否生效。评分超过 5 时，后端应拒绝请求，并且评论数不应增加。

#### 1. 查询测试前评论数
```powershell
$beforeDish = Invoke-RestMethod `
  -Method Get `
  -Uri "$BASE/api/dishes/1"

$beforeReviewCount = $beforeDish.data.base.reviewCount
$beforeReviewCount
```

#### 2. 提交非法评分
$badReviewBody = @{
    rating = 6
    content = "非法评分测试，rating 超过 5。"
    imageUrl = "https://example.com/bad-review.jpg"
} | ConvertTo-Json

$badResp = Invoke-WebRequest `
  -Method Post `
  -Uri "$BASE/api/dishes/1/reviews" `
  -Headers $normalHeader `
  -ContentType "application/json" `
  -Body $badReviewBody

$badJson = $badResp.Content | ConvertFrom-Json

"HTTP Status = $($badResp.StatusCode)"
"Business code = $($badJson.code)"
"Message = $($badJson.message)"

if ($badJson.code -eq 400) {
    Write-Host "[PASS] 非法评分被拒绝，数据校验生效"
} else {
    Write-Host "[FAIL] 非法评分不应被接受"
    $badJson | ConvertTo-Json -Depth 10
}
```
预期结果：
```text
HTTP Status = 200
Business code = 400
Message = rating: 最大不能超过5
[PASS] 非法评分被拒绝，数据校验生效
```

#### 3. 再次查询评论数
```powershell
$afterDish = Invoke-RestMethod `
  -Method Get `
  -Uri "$BASE/api/dishes/1"

$afterReviewCount = $afterDish.data.base.reviewCount

"beforeReviewCount = $beforeReviewCount"
"afterReviewCount  = $afterReviewCount"
```
预期结果：
```text
beforeReviewCount = 1
afterReviewCount  = 1
```
说明非法评分没有写入评论表，也没有影响菜品评分统计。
