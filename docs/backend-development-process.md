# 后端开发流程记录

本文档用于课程报告和答辩材料整理，记录后端从接口约定、数据库建模到联调测试的主要开发过程。日期按阶段整理，便于说明项目是逐步推进完成的；如需放入最终报告，可结合实际提交记录再微调表述。

## 2026-05-20：需求拆分与接口边界

本阶段主要把移动应用评分表中的要求拆成后端需要支撑的能力，并和 Android 端约定数据交换方式。

完成内容：

- 明确后端负责用户注册登录、食堂与菜品数据、评论评分、收藏、补充菜品、管理员审核、公告等能力。
- 约定统一响应格式，使用 `code/message/data` 返回业务结果。
- 约定登录后使用 JWT，前端在请求头中携带 `Authorization: Bearer token`。
- 编写前端接口契约文档，方便前端在后端未完全实现时先按接口开发页面。

设计考虑：

- 大作业验收更重视功能闭环，因此接口优先覆盖主要用户流程。
- 列表接口先不做复杂分页，避免在演示数据量较小时增加额外复杂度。
- 后端返回字段尽量直接服务 Android 展示，例如菜品评分、评论数、标签、图片地址等。

## 2026-05-22：数据库结构与基础工程

本阶段建立 Spring Boot 后端工程和 MySQL 数据库结构。

完成内容：

- 搭建 Spring Boot 3 后端项目。
- 接入 MyBatis-Plus，简化实体表的基础增删改查。
- 建立 MySQL 数据库脚本 `schema.sql`。
- 设计核心数据表：
  - 用户表
  - 食堂表
  - 窗口表
  - 菜品表
  - 菜品标签表
  - 评论表
  - 收藏表
  - 补充菜品提交表
  - 公告表
  - 拥挤度上报表

设计考虑：

- 食堂、楼层、窗口、菜品分层建模，方便前端按食堂和窗口筛选。
- 标签独立存储，方便后续筛选、展示和推荐。
- 用户上传的新菜品先进入提交表，由管理员审核后再写入正式菜品表。

本阶段实现用户身份相关功能。

完成内容：

- 实现用户注册接口。
- 实现用户登录接口。
- 使用 Spring Security 和 JWT 保存登录态。
- 区分普通用户和管理员角色。
- 添加管理员接口访问控制。

设计考虑：

## 2026-05-26：食堂、窗口、菜品与标签接口

本阶段实现应用的主体浏览能力。

完成内容：

- 实现食堂列表接口。
- 实现食堂详情接口，包含楼层和窗口数据。
- 实现菜品列表接口。
- 实现菜品详情接口。
- 支持按食堂、窗口、关键词和标签筛选菜品。
- 实现标签列表接口，供前端上传菜品和偏好选择使用。

设计考虑：

- 食堂列表即使某些食堂暂时没有菜品，也应返回完整食堂数据。
- 没有数据的食堂或楼层由前端显示“暂无数据”，后端保持结构完整。
- 菜品价格为 `0` 时表示称重计价，前端展示为“称重计价”。

## 2026-05-28：评论、评分、收藏与互动

本阶段补齐用户对菜品的反馈能力。

完成内容：

- 实现菜品评论接口。
- 实现五颗星评分写入。
- 统计菜品平均评分和评论数。
- 实现收藏和取消收藏。
- 实现评论点赞、点踩和取消状态。

设计考虑：

- 收藏数据与登录用户绑定，方便个人页展示。

## 2026-05-30：补充菜品与管理员审核

本阶段实现用户上传新菜品、管理员审核的闭环。

完成内容：

- 实现用户提交新菜品接口。
- 支持提交菜名、食堂、窗口、价格、描述、辣度、标签和图片地址。
- 实现管理员查看待审核菜品接口。
- 实现管理员批准和驳回接口。
- 审核通过后将提交内容写入正式菜品表。

设计考虑：

- 普通用户提交的数据不直接进入正式菜品列表，避免错误数据污染展示。
- 管理员审核时需要看到足够详细的信息，包括图片、标签和描述。
- 审核流程保持简单，适合课程验收演示。

## 2026-06-01：真实数据整理与种子数据

本阶段把真实食堂和菜品数据导入数据库。

完成内容：

- 根据共享表格整理食堂、楼层、窗口、菜品、标签和图片文件名。
- 生成 `seed.sql`，包含真实食堂、窗口、菜品和初始评论。
- 导入食堂封面图和菜品图。
- 添加初始评分和评论，让首页、详情页和列表页有完整展示效果。

设计考虑：

- `schema.sql` 负责建表，`seed.sql` 负责演示数据，二者分开便于重复初始化。
- 图片统一放在后端 `uploads` 目录，通过静态资源路径返回给 Android。
- 中文 SQL 导入时必须使用 `--default-character-set=utf8mb4`，避免乱码。

## 2026-06-03：图片上传与真机联调

本阶段解决移动端真实上传图片和真机访问后端的问题。

完成内容：

- 新增图片上传接口 `POST /api/uploads/images`。
- 使用 multipart 表单上传图片文件。
- 后端保存图片到 `uploads/dishes`。
- 返回图片访问地址，供菜品提交和审核展示使用。
- 配置 multipart 上传大小限制，单张图片最大 10MB。
- Android 端使用 `127.0.0.1:8080` 配合 `adb reverse` 访问电脑后端。

设计考虑：

- 真机不能直接访问电脑上的 `localhost`，使用 `adb reverse` 可以减少校园网和电脑 IP 变化带来的问题。
- 上传图片保存在后端本地目录，满足课程验收和小规模演示需求。
- 目前不引入对象存储，避免部署和权限配置过重。

## 后端整体架构

后端采用分层结构：

- `controller`：接收 HTTP 请求，返回统一 API 响应。
- `service`：处理业务逻辑，例如审核通过、评分统计、收藏切换。
- `domain/entity`：数据库实体。
- `domain/mapper`：MyBatis-Plus 数据访问接口。
- `dto`：前后端传输对象。
- `config`：安全、静态资源、上传大小等配置。
- `resources/db`：建表脚本和种子数据。

整体数据流：

```text
Android 页面
  -> Retrofit API 请求
  -> Spring Controller
  -> Service 业务处理
  -> Mapper 访问 MySQL
  -> DTO 组装展示字段
  -> Android 渲染列表、详情、审核页
```
